# プログラミング言語メタ理論のための次世代Ottツールチェーン：Rust、WebAssembly、およびTypstによる完全なアーキテクチャ再構築と詳細設計

## 序論：プログラミング言語意味論におけるレガシーOttの制約と近代化の必然性
プログラミング言語の構文および意味論の定義は、言語設計、型システムの証明、およびコンパイラの実装において極めて重要な基盤となる。この領域において、Ott（オット）は、非形式的な数学的記述に近い簡潔で可読性の高いASCII記法を用いて言語の構文と意味論を定義し、それを多様なターゲットへとコンパイルするメタ言語およびツールとして長年にわたりデファクトスタンダードの地位を確立してきた 。従来のOttツールはOCamlで実装されており、定義された入力ファイルから、高品質な組版のためのLaTeXソースコード、形式的定理証明支援系であるCoq、HOL、Isabelle/HOL用の定義、意味論の大規模モデリングに向けたLem、さらにはプログラム実装に向けたOCamlの抽象構文木（AST）およびMenhirパーサーを自動生成する機能を有する 。このツールは、Lightweight JavaやOCamlのコアフラグメント、POPLmarkのF<:言語など、大規模かつ複雑なプログラミング言語の厳密な意味論定義に用いられてきた 。
しかしながら、現行のOttアーキテクチャはいくつかの構造的およびエコシステム上の限界を抱えている。第一に、組版ターゲットがLaTeXに強く依存している点である。LaTeXは極めて強力な組版システムであるが、コンパイル速度の遅さ、複雑なマクロの依存関係、およびエラーメッセージの難解さが、言語設計の迅速な反復プロセス（ラピッドプロトタイピング）を阻害する要因となっている。第二に、ツール自体がOCamlに依存しており、`ocamlgraph`などの外部パッケージを用いた依存関係解決や位相ソートに依存しているため、他のモダンなツールチェーンへの組み込みやブラウザベースの環境での実行が困難である 。
本報告書は、これらの課題を根本的に解決するため、Ottのすべての詳細な機能をRust言語を用いて完全に再実装し、WebAssembly（WASM）を介して次世代の高速組版システムであるTypstに直接統合するための包括的なアーキテクチャプランと詳細な設計を提示する。TypstはRustで記述されたモダンなマークアップ言語であり、リアルタイムに近いコンパイル速度、純粋関数型のスクリプト環境、およびWASMプラグインのネイティブサポートを提供する 。OttのコアロジックをRustベースのWASMプラグインとしてTypstに埋め込むことで、研究者はLaTeXのコンパイルオーバーヘッドから解放され、Typst上で直接、推論規則や構文規則のリアルタイムレンダリングを確認しながら言語設計を行うことが可能となる。本設計は、Ottのユーザマニュアルに規定されたすべての機能（メタ変数、サブルール、リスト形式、コンテキストルール、束縛仕様、準同型写像など）を網羅し 、既存のOttと完全な互換性を持つツールチェーンを構築するための道筋を詳細に論じる。

## 抽象構文木（AST）とデータモデルのRustへの完全なマッピング
Ottのパーサーは、メタ変数、終端記号、非終端記号、およびそれらから構成される生成規則と意味論的推論規則を含むASCII仕様を読み込み、内部的に複雑な抽象構文木（AST）を構築する 。OCamlの実装では、これらの要素は代数的データ型（バリアント）を用いて表現されている。Rustへの移行にあたっては、Rustの強力な`enum`システムを利用してこれらの構造を再現するが、Rustの厳密な所有権モデルとメモリ管理の制約に適応させるための綿密な設計が要求される。

### メタ変数、終端記号、および非終端記号のモデリング
プログラミング言語の仕様は、文法規則の集合として定義される。その基礎となるのが、具体的な構文トークンを示す「終端記号（Terminals）」、変数や識別子の集合を表す「メタ変数（Metavariables、例：`x`、`y`）」、および他の文法生成規則への参照を示す「非終端記号（Non-terminals、例：`expr`、`typ`）」である 。従来のOttはこれらを解析し、メタ変数の命名規則や判断形式の一貫性を型チェック（ソートチェック）によって保証する 。
Rust実装では、これらの基底要素を再帰的なASTノードとして表現する。相互に再帰的なAST構造は、Rustにおいてコンパイル時のサイズ決定が不可能であるため、`Box<T>`や、パフォーマンス最適化のためのアリーナアロケータ（`bumpalo`クレートなど）を用いたメモリ確保戦略が不可欠となる。以下の表は、OCamlの主要なデータ構造をRustのモデルへとマッピングした設計である。

| Ottの基本概念 (OCaml表現) | 提案されるRustのデータ構造 | 構造的役割および機能的意味合い |
| --- | --- | --- |
| metavar | enum Element { MetaVar(String) } | 代入や束縛の対象となる識別子。構文木の末端として機能し、後に束縛変数へと解決される。 |
| terminal | enum Element { Terminal(String) } | 構文的に固定されたトークン（例：+、lambda、::=）。言語の具象構文を形成する。 |
| nonterm | enum Element { NonTerminal(String) } | 他の生成規則を参照する要素。型の帰納的定義を構成する。 |
| rule | struct Rule { name: String, productions: Vec<Production> } | 構文規則または意味論的判断の全体を定義するコンテナ。 |
| prod | struct Production { elements: Vec<Element>, homs: Vec<Hom> } | 規則内の個別のケース。要素の列と、ターゲット言語への変換を定義する準同型写像（Hom）を保持する。 |

### サブルール、コンテキスト規則、および依存関係グラフの位相ソート
Ottの極めて強力な機能の一つに、サブルール（Subrules）やコンテキスト文法（Context grammars）、およびリスト形式のサポートがある 。たとえば、値（values）の文法を式（expressions）の文法の厳密な部分集合として定義するサブルール機能では、ツールが自動的にそれらの包含関係を解析し、CoqやIsabelleの証明支援系において部分集合であることを判定する述語（例：`is_v v'`）や型強制の関数を生成する 。
レガシーなOCamlアーキテクチャでは、これらの型や規則間の相互再帰的な依存関係を解決し、Coqのような厳密な順序付けを要求する定理証明支援系向けにコードを出力するために、`ocamlgraph`ライブラリを用いて依存関係グラフの構築と位相ソート（Topological sort）を行っている 。Rustへの移行においては、このコア依存関係を`petgraph`クレートに置き換える。構文解析フェーズの直後に、Rustエンジンはすべての非終端記号と規則をノードとする有向非巡回グラフ（DAG）を構築する。このグラフ解析により、サブルールの定義が構造的に妥当であるか（循環定義がないか）を検証し、定理証明支援系向けに出力する際の正しい宣言順序を決定する。
さらに、評価文脈（Evaluation contexts）を定義するためのコンテキスト規則は、項の中に単一の「穴（hole）」を持つ構文要素として解析される。RustのASTでは、これを`Context(Box<Element>)`として特別なノード型として扱い、後にプラグインの置換アルゴリズム（substitution logic）と連携して、穴への項の代入操作を自動生成するためのメタデータとして保持する。

## 構文解析エンジンの再構築とパーサー生成ロジック
Ottは、ユーザーが定義した任意の文脈自由文法（Context-Free Grammar）を即座にパーサーとして構築し、それを用いて意味論の推論規則や、ドキュメント内のシンボリックな項（Symbolic terms）をパースする能力を持つ 。OCaml環境では、この動的解析は通常、Menhirやocamlyaccのメカニズムを応用して構築されるが、Rust環境では異なるアプローチが必要となる。

### 動的パーサーと曖昧性の解決（Parsing Priorities）
ユーザーが定義したASCII記法から直接ASTを構築するために、Rust実装では`bnf`クレートや`LALRPOP`などの強力なパーサーコンビネータまたはパーサージェネレータを活用する 。Ottのユーザマニュアル第17章に規定されている「Parsing Priorities（パースの優先順位）」は、演算子の結合順序や優先度を指定することで構文の曖昧性を排除する重要な機能である 。
Rustベースの構文解析エンジンは、まず言語定義ファイル（例：`t ::= t1 + t2`、`t ::= t1 * t2`）を読み込み、それに付随する優先順位のアノテーションを解析する。その後、Rustエンジン内部で動的にLALR(1)状態機械またはEarleyパーサーを構築し、ユーザーが記述した推論規則の前提（Premises）や結論（Conclusion）の項をパースする。Ottの実験的機能である「同一のコード生成をもたらす複数のパース結果の許容（`-picky_multiple_parses <true>`）」も、Earleyパーサーの曖昧性許容モードを活用することで、Rust上で高精度に再現することが可能である 。パースエラーが発生した場合には、Typstのコンパイルエラーとして即座にユーザーにフィードバックされるため、LaTeX環境に比べて圧倒的に優れた開発者体験を提供する。

### ターゲット言語向けのパーサー自動生成機能
Ottは、定義された言語の構文から、その言語自体の実装に利用できるMenhirパーサー（`.mly`ファイル）とOCamlのボイラープレートコードを実験的に生成する機能を持つ 。この機能をRustアーキテクチャに移植する際、エンジンは構築した動的ASTを解析し、Menhir向けの文法定義ファイルを出力するジェネレータを実装する。同時に、現代のプログラミング言語実装のトレンドに合わせて、Rust製のパーサー（例えば`nom`や`chumsky`ベースのパーサー）のボイラープレートコードを自動生成する新たなターゲットバックエンドを追加することも、本再構築における強力な拡張機能として機能する。

## 束縛仕様（Binding Specifications）と代入アルゴリズムの深層
Ottの中核的な価値の一つは、言語の変数束縛に関する複雑な規則を簡潔に指定し、自由変数（Free variables）の計算、単一代入（Single substitution）、複数代入（Multiple substitutions）、および束縛補助関数を自動生成する機能である 。これらのアルゴリズムをRustで完全に複製し、かつ最適化することは、本プロジェクトにおける最も技術的に高度な課題である。

### 具象的束縛と局所名無し表現（Locally-Nameless Representation）
従来のOttはデフォルトで「完全な具象的（Fully concrete）な束縛表現」をサポートしている。この手法では、変数は証明支援系の中で文字列や自然数として直接表現され、アルファ同値性（Alpha equivalence）による商を取らない 。このアプローチは多くの単純なケースで機能するものの、ラムダ計算のような言語の形式化においては、ユーザー自身でキャプチャ回避代入（Capture-avoiding substitution）の補題を証明しなければならないという重い負担を強いる。
この問題を解決するため、Ottには実験的な「局所名無し表現（Locally-nameless representation）」のバックエンドが存在する 。局所名無し表現は、束縛変数にド・モルガン・インデックス（de Bruijn indices）を使用し、自由変数には名前（文字列）を保持するハイブリッドアプローチである。これにより、項の表現が一意になり、アルファ変換の問題が本質的に回避される 。
Rust実装においては、この局所名無し表現を第一級市民として扱う。具体的には、Rustのエコシステムに存在する`unbound`クレートなどの局所名無し表現ライブラリの概念を応用する 。Rustエンジン内でASTを生成する際、Ottの仕様における`bind binders(p) in t'`のような束縛アノテーションを検知した時点で 、対象となる変数を`Bind<P, T>`のようなRustのネイティブな束縛構造体でラップする。これにより、Rustのコンパイラプラグイン内でASTのアルファ同値性判定やキャプチャ回避代入が自動的に導出（`#`）され、定理証明系向けのコードを生成する際の論理的な正当性がシステムレベルで担保される。

### 自由変数と代入関数の自動生成メカニズム
代入関数の生成アルゴリズムは、4つの基本的な公理（変数自身への代入、異なる変数への代入、関数適用における代入、および抽象における代入）に基づいて帰納的に定義される 。Rustベースのコードジェネレータは、構文解析されたASTを深さ優先探索（DFS）で巡回し、ターゲット言語（Coq、HOL、Isabelle）向けの再帰的な代入関数を構築する。
Ottは、パターンのバインダーを収集するユーザー定義の補助関数（例：`binders(p)`）をサポートしている 。Rustエンジンは、これらの補助関数を構文木上のメタ変数の集合を返す関数としてモデル化し、ターゲットコード上で変数集合の和集合（Union）やリスト上のマッピング関数（`list_flat_map`など ）として展開するロジックを実装する。たとえば、Coq向けに出力する場合、相互に再帰的な複数の代入関数群を`Fixpoint`と`with`キーワードを用いて出力し、変数の衝突を回避するための条件分岐（`if string_dec x y then... else...`）を自動的に挿入する。

## 準同型写像（Homomorphisms）による多ターゲット変換論理
Ottのもう一つの基幹技術が準同型写像（Homomorphisms、通称homs）である。これは、特定の構文生成規則や定義が、LaTeX、Coq、HOL、Isabelle、OCamlといった各ターゲット出力フォーマットにおいてどのように表現されるべきかをユーザーが細かく制御するためのメタアノテーションである 。

### Homの構造とRustでのテンプレート解決
Homは、生成規則に直接付与されるか、あるいは別の「Homブロック」として一括して定義される 。例えば、Coqにおいて自然数として表現したいメタ変数は`{{ coq nat }}`と記述され、LaTeXで特定の数式マクロを展開したい場合は`{{ tex \frac{[[t1]]}{[[t2]]} }}`のように記述される。ここで`[[t1]]`のような抽出変数（Extraction variables）は、構文木の子ノードを再帰的に評価した結果を文字列として埋め込むことを意味する 。
Rustエンジンでは、このHomシステムを強力なテンプレートエンジンとして再構築する。各ASTノードが特定のバックエンド向けにコード生成を要求された際、ジェネレータはまずそのノードに紐づくHom文字列を検索する。存在する場合、抽出変数の構文を正規表現または専用のパーサーで抽出し、子要素の生成結果を再帰的に取得してテンプレートに埋め込む。

| ターゲット | 従来のOtt構文例 | Rust/Typstアーキテクチャにおける処理戦略 |
| --- | --- | --- |
| LaTeX | {{ tex \lambda }} | 廃止またはTypst記法への変換。Typst向けには {{ typ $lambda$ }} などの新しいHomタグを導入するか、レガシーサポートとしてTeXからTypstへの変換レイヤー（mitexの応用）を挟む。 |
| Coq | {{ coq nat }} | Rustジェネレータが直接 .v ファイルに nat という文字列を出力する。 |
| Isabelle | {{ isasyn [[t]]\<bullet>[[t']] }} | IsabelleのMixfix構文（isasyn）を解釈し 、.thy ファイルの datatype 宣言等に正確にマッピングする。 |
| OCaml | {{ ocaml int }} | ASTジェネレータが .ml ファイルの型定義に int を使用する。 |
| Identity | {{ ich [[t]] }} | 括弧の除去など、内部的なAST変換（Identity translation）をRust内で直接処理し、他のすべてのバックエンドの出力に影響を与える 。 |
リスト形式（List forms）のHomも極めて重要である。Ottは`[[x1 t1.. xn tn]]`のような記法を用いて、レコード型や関数の引数リストを表現する。Rustジェネレータは、このドット表現（`..`）を解析し、対象のリストがターゲット言語（例えばIsabelleの`(ident * t) list`）においてどのように構造化されるべきかを判断し、`List.rev`などのリスト操作関数を含む対応するコードを生成する 。

## Typst WebAssemblyプラグインプロトコルの統合
本プロジェクトの最大の革新は、OCamlからRustへの単なる移植にとどまらず、WebAssemblyを介したTypstへのネイティブな統合である。Typstのプラグインシステムは、Rustなどの言語で記述されたコードを32ビットの共有WASMライブラリとしてコンパイルし、サンドボックス化された環境で実行する機能を提供する 。これにより、ドキュメントのコンパイルと同時に言語定義のパースとレンダリングが行われる。

### WASM ABIとメモリバッファの制約
Typstのプラグインプロトコルは非常に低レベルであり、引数として複数のバイトバッファ（メモリ長の32ビット整数）を受け取り、結果を単一のバイトバッファとして返す形式を要求する 。セキュリティ上の理由からWASI（WebAssembly System Interface）へのアクセスは制限されており、ファイルシステムからの読み書きや標準出力はサポートされていない 。さらに、プラグインの関数は「純粋（Pure）」であることが求められ、同じ引数に対しては常に同じ結果を返し、観測可能な副作用を持たないことが前提となる（状態保持のための`plugin.transition` APIは存在するが、Ottの構文解析においては純粋関数としての実装が適している） 。
このバイトレベルのやり取りを隠蔽し、Rust側で安全かつ簡潔にロジックを実装するために、本アーキテクチャでは`wasm_minimal_protocol`クレートを採用する 。このクレートが提供する`#[wasm_func]`マクロを利用することで、Rust関数とTypstの間のポインタ演算とメモリ割り当てが自動化される。

### 直列化（Serialization）とWASM境界の突破
TypstのWASMプラグインの最大の技術的障壁は、プラグインから直接Typstのネイティブな`content`型（数式やテーブルなどのレンダリングオブジェクト）を返すことができない点である 。WASMプラグインは単なるバイト配列しか返せないため、Rustで計算したOttの推論規則や構文木の構造を、Typst側で解釈可能な形式に直列化して渡す必要がある。
この問題を解決するため、`sertyp`および`typwire`などのパッケージが提供するCBOR（Concise Binary Object Representation）エンコーディングのアプローチを統合する 。処理のパイプラインは以下の通りとなる。

1. **入力フェーズ**: ユーザーがTypstドキュメント内にOtt仕様を記述し、Typstスクリプトがそれを文字列としてWASMプラグインに渡す。
2. **Rust解析フェーズ**: WASM内で実行されるRustエンジンが文字列を解析し、言語のASTを構築。サブルールや束縛情報を解決し、前述のHomテンプレートを適用して、画面に描画されるべき要素の抽象的な表現を構築する。
3. **直列化フェーズ**: Rustエンジンが、構築された描画要素のツリーをCBOR形式のバイト配列に直列化する。このCBORデータには、ルールの名前、前提条件のリスト、結論、および数式の構成要素が含まれる 。
4. **Typst復元フェーズ**: バイト配列がTypst側に返却される。Typstスクリプトは`typwire.cbor.decode()`などを用いてバイト配列をTypstの辞書（Dictionary）や配列（Array）に復元する 。
5. **レンダリングフェーズ**: 復元された辞書データをもとに、Typstの再帰関数がネイティブな組版コマンドを呼び出し、実際のドキュメントコンテンツを生成する。

## Typstフロントエンドにおける組版ロジックの構築（LaTeXからの脱却）
従来のOttは、複雑なLaTeXマクロを自動生成し、LaTeX文書内でそれらを呼び出すことで推論規則（Inference rules）や文法（Grammars）の組版を実現していた 。Typst環境では、これらの組版処理をモダンなTypstパッケージを組み合わせて実現する。

### 推論規則と証明木のレンダリング（currystパッケージの応用）
言語の意味論における型付け規則や操作的意味論のステップは、証明木（Proof trees）として描画される。LaTeXにおける`mathpartir`パッケージの代替として 、Typstでは`curryst`パッケージを活用する 。
CBORから復元された推論規則の辞書データは、Typstのラッパースクリプト内で`curryst`の`prooftree`および`rule`関数にマッピングされる。例えば、結論が1つで前提が複数ある規則は以下のように動的に展開される。

```代码段
#prooftree(
  rule(
    name: "T-Abs",
    $Gamma tack lambda x : tau_1.e : tau_1 -> tau_2$, // 結論
    $Gamma, x : tau_1 tack e : tau_2$ // 前提
  )
)

```
`curryst`パッケージは、`min-premise-spacing`（前提間の最小スペース）、`title-inset`（横線の延長量）、`stroke`（線のスタイル）などの高度なカスタマイズオプションを提供しており、これらをTypst側の関数引数として露出させることで、ユーザーはLaTeX環境以上の柔軟性を持って推論規則のレイアウトを調整することが可能となる 。複数のルールを並べて表示する場合には、`rule-set`関数を利用し、`column-gutter`や`row-gutter`によってグリッド状の配置を行う 。

### 文脈自由文法の組版とBNF記法のレンダリング
プログラミング言語の構文定義は、Backus-Naur Form（BNF）形式で表現される。LaTeXで`tabular`環境を用いていたレイアウトは、Typstでは`simplebnf`パッケージの概念を取り入れたネイティブなGridやTable構造へと変換される 。
Rustから渡された文法規則データに従い、Typstスクリプトは左側に非終端記号、中央に`::=`、右側に`|`で区切られた生成規則を整列させる。Ott特有の機能であるコメントの出力（`{{ com types: }}`など）についても 、右端の列に自動的にフローティングさせて配置するロジックを実装し、美しい言語仕様書の自動生成を実現する。

### フィルターモードとシンボリックな項の埋め込み
Ottの非常に強力な機能として、LaTeXやCoqのソースコード内に埋め込まれたシンボリックな項（例えば `[[ \x.x x ]]`）を検出し、パースして型チェックを行った上で組版されたコードに置換する「フィルター機能（Filter mode）」が存在する 。
TypstとRustの統合環境においては、外部のプリプロセッサを実行する必要はなくなる。Typstの強力な`#show`ルールと正規表現マッチング機能を活用することで、ドキュメントのコンパイル時に動的フィルタリングを実現する 。実装としては、`mitex`パッケージがLaTeXコードをWASM経由でTypstの数式コンテンツに変換するアーキテクチャを踏襲する 。
ユーザーがドキュメント内で特定のデリミタ（例：`[[... ]]`）を用いた場合、Typstの`#show`ルールがそのテキストを捕捉し、WASMプラグインに渡す。Rustエンジンは定義されたOttの文法に従ってその文字列をパースし、構文木が正当であれば、対応するTypstの数式（Math mode）文字列またはCBORデータを返す。Typstの数式モード（`$... $`）は、シンボル（`#sym.pi`など）やスペースの自動調整をネイティブにサポートしているため 、LaTeXのような複雑なマクロ定義を必要とせず、簡潔かつ美しい数式描画が達成される。

## 定理証明支援系および実装系へのコード生成アーキテクチャ
Typstプラグインはドキュメントの組版においては極めて優れているが、サンドボックス化されたWASM環境からはローカルファイルシステムに直接ファイルを書き出すことができないという制約がある 。したがって、Ottの本来の目的である「証明支援系コードの生成」を達成するためには、このシステムをデュアルモードアーキテクチャとして提供する必要がある。すなわち、ドキュメントのレンダリングに用いる「WASMプラグインモード」と、外部ツールチェーンとして各種ソースファイルを生成する「ネイティブCLIモード」である。両者は完全に同一のRustコア（パーサーおよびASTジェネレータ）を共有する。

### Coq向け定理証明コードの生成
Coqへのコンパイルは、Ottの最も利用されるバックエンドの一つである 。RustエンジンのCoqジェネレータは、ASTを走査し、非終端記号をCoqの`Inductive`データ型に変換する。相互に再帰的な定義は、位相ソートの結果に基づいて`with`句を用いて出力される。
さらに、Ottが提供するリスト形式のサポートには、Coq側に膨大な証明基盤（`ott_list_base.vo`、`ott_list_predicate.vo`、`ott_list_takedrop.vo`など）が必要である 。Rustアーキテクチャでは、これらの標準的なリスト操作の補題をあらかじめライブラリとして内包し、ユーザーの言語定義に応じて必要な補題のインスタンス化コード（例えば`ott_list_flat_map`など ）を自動生成して`.v`ファイルに出力する。

### Isabelle/HOLおよびHOL4向けコードの生成
Isabelle/HOLバックエンドでは、Isabelle特有のMixfix構文（`isasyn`アノテーション）の処理が重要となる 。Rustジェネレータは、`{{ isasyn [[t]]\<bullet>[[t']] }}`のようなHomアノテーションを解析し、Isabelleの`datatype`宣言や`inductive`宣言において正確な構文定義を出力する 。これにより、Isabelleの証明スクリプト内で、Ottで定義した通りの数学的記法を用いて直接証明を記述することが可能となる。
HOL4バックエンド（`.sml`ファイル出力）についても同様に、ASTから直接Standard ML形式の定義を出力し、関係の推移的閉包やリスト操作の補題を組み込むロジックを実装する 。

### OCaml実装およびMenhirパーサーの生成
言語設計のプロトタイピングにおいて、定義した言語のインタープリタを即座に記述できることは極めて重要である。OttはOCamlのAST定義と、それをパースするためのMenhirパーサーを生成する 。
ネイティブCLIモードのRustエンジンは、メタ変数や規則の依存関係グラフを走査し、OCamlの`type`宣言群（バリアント型）を生成する。さらに、Ottの構文優先順位（Parsing Priorities）に関する指定（Section 17）を読み解き 、Menhirの`.mly`ファイルにおける`%left`や`%right`といった結合性と優先順位のディレクティブへと正確にマッピングする。これにより、定理証明系に渡される構文と完全に一致するパーサーが、実装向けにも自動生成されることが保証される。

## 実装計画とロードマップ
この極めて複雑なアーキテクチャを実現するためには、段階的かつ体系的な実装アプローチが不可欠である。以下の4つのフェーズに基づくロードマップを提示する。

### フェーズ1：コア解析エンジンとAST構築（Rust CLI基盤）
最初のフェーズでは、WASMの複雑性を排除し、純粋なRustアプリケーションとしてOttの構文解析コアを構築する。

- **パーサーの実装**: `bnf`クレートや`LALRPOP`を用いて、Ott自身のメタ言語構文（ASCII記法）を解釈するレキサーおよびパーサーを開発する 。Ottユーザーマニュアルの第2章に準拠した入力を完全にパースできるようにする 。
- **データモデルの確立**: ASTノード（`Rule`, `Production`, `Element`など）の定義と、相互再帰的な型を管理するための参照構造（Box化）を実装する。
- **依存関係解決**: `petgraph`を導入し、サブルールや定義間の依存関係グラフを構築、位相ソートによる循環参照の検出を実装する 。
- **セマンティックチェック**: メタ変数の整合性や、判断形式（Judgements）の正しい使用に関する検証アルゴリズムを実装する 。

### フェーズ2：束縛仕様の解決とコードジェネレータの実装
ASTが確立された後、Ottの最重要機能である代入アルゴリズムと準同型写像の処理を組み込む。

- **束縛仕様の評価**: 「具象的束縛」の処理ロジックを実装し、続けて`unbound`クレートを活用した「局所名無し表現」のためのインデックスシフトとアルファ変換機能を追加する 。
- **補助関数の生成**: ASTを巡回し、自由変数（`freevar`）や代入（`subst`）関数のコードを再帰的に合成するロジックを開発する 。
- **Homテンプレートエンジン**: `{{ coq... }}`や`{{ tex... }}`ブロックを解析し、抽出変数（`[[t1]]`など）をASTの子要素の生成結果で置換する文字列置換エンジンを構築する 。

### フェーズ3：Typst WASMプロトコルの統合と組版ロジック
このフェーズにおいて、RustコアをTypstに統合し、組版ツールとしての機能を完成させる。

- **WASM ABIの適応**: `wasm_minimal_protocol`クレートを導入し、Typstからの呼び出しインターフェース（バイトバッファのやり取り）を確立する 。
- **CBOR直列化**: `sertyp`または`typwire`を用いて、解決済みのASTおよび組版用データをCBOR形式に直列化してWASM境界を越える仕組みを構築する 。
- **Typstパッケージの開発**: デコードされたCBOR辞書を読み込み、`curryst`（推論規則用）および`simplebnf`のレイアウト（文法用）に変換して描画するTypst関数群（`.typ`ファイル）を作成する 。
- **フィルターモードの実現**: `mitex`のパラダイムに倣い、Typstの`#show`ルールを利用してドキュメント内のシンボリックな項を捕捉し、WASM経由でリアルタイムに数式コンテンツへ変換する機能を実装する 。

### フェーズ4：形式証明系バックエンドの完成と最適化
最終フェーズでは、外部ファイル生成機能を完成させ、すべてのOtt機能の複製を完了する。

- **Coq / Isabelle / HOL出力**: 各定理証明支援系向けのファイル生成ルーチンを完成させる。IsabelleのMixfix構文や、Coqのリスト補助定理ライブラリとの連携を精緻化する 。
- **実装言語（OCaml/Rust）およびパーサー出力**: ASTからプログラム実装向けのボイラープレートコードとMenhir用パーサー定義を生成する機能を組み込む 。
- **パフォーマンステストと検証**: 既存のOtt仕様（TAPLフラグメント、Lightweight Javaなど ）を新システムにロードし、従来のOCaml実装と出力が論理的に一致することを検証する。

## 結論
レガシーなOttアーキテクチャは、プログラミング言語意味論の厳密な定義と定理証明系への連携において不可欠な役割を果たしてきた。しかし、OCamlおよびLaTeXという基盤技術への依存は、コンパイル速度、エラー追跡、および現代的なウェブファーストな文書作成環境との統合において限界を見せている。
本報告書で提示した、Rust、WebAssembly、およびTypstを組み合わせた次世代アーキテクチャは、これらの課題を根本から解決する。`petgraph`による位相ソート、`unbound`クレートを用いた局所名無し表現のネイティブサポート、およびCBOR直列化を通じたTypstフロントエンドとの通信というモダンなRustの強力なエコシステムを活用することで、従来のOttが持っていた束縛仕様、サブルール、コンテキスト規則、そして推論規則の自動生成といった高度なメタ理論的機能を一切損なうことなく再構築することが可能である。さらに、Typstの`curryst`パッケージや`#show`ルールを用いた即時プレビュー可能な組版機能は、言語設計者に対し、かつてないほど迅速で直感的な反復プロセスを提供する。このアーキテクチャシフトは、単なるツールの移植にとどまらず、プログラミング言語研究の生産性と正確性を飛躍的に高めるパラダイムシフトとなる。
